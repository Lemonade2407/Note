# 架构分析和设计
- 架构设计->详细设计
## 架构设计
### 体系结构设计
- 体系结构设计是包括硬件、网络、软件在内的整体方案
- 架构模式设计只包含软件
- ![[体系结构设计.png]]
#### 常见架构模式
[10 种常用的软件架构模式](https://zhuanlan.zhihu.com/p/37590419)
- 分层模式：
	- 具有相同功能的组件将被组织成水平层。 
	- 每一层在应用程序中都扮演着特定的角色。
	- ![[分层架构模式.png]]
- MVC模式：
	- M模型-包含应用程序的数据。 
	- V视图-显示部分底层数据并与用户交互。 
	- C控制器-在模型和视图之间进行业务处理。
- 客户端-服务器模式（Client/Server，C/S）
	- 服务器：服务提供者 
	- 客户端：服务请求者 
	- 划分任务/工作负载
- 浏览器-服务器模式（Brower/Server，B/S）
	- B/S简化了客户端，它无需像C/S模式那样在不同的客户机上安装不同的客 户应用程序，而只需安装通用的浏览器软件。
- 点对点模式 P2P
	- 对等点既可以作为客户端向其他对等点请求服务，也可以作为服务器向其他对等点提供服务 
	- 文件分享网络、区块链
- 事件驱动模式（事件总线模式）
	- 高度可扩展的响应式应用程序
	- 异步传递和处理事件
	- ![[事件驱动.png]]
- 微服务模式
	- 故障隔离
	- 技术异构
	- ![[微服务模式.png]]
### 功能模块结构设计
- 设计软件的模块及交互关系。 
- 设计软件的接口。 
- 设计软件的数据模型。
- ![[功能模块设计.png]]
#### 功能模块设计的原则
- 模块化：大问题分解为小问题。 
- 高内聚、低耦合：
	- 高内聚：一个类 / 模块只干一件事，功能高度集中。 
	- 低耦合：模块之间依赖少、牵连浅、互不打扰。 
- 逐步落地：按抽象级别从高到低进行软件设计。 
- 一致性：各个模块的术语和符号一致，模块之间的接口应保持一致。
#### 功能模块的划分
![[功能模块图.png]]
- 模块的相对独立性：
	- 模块只具有相对单一的功能且与其他模块没有太多的联系。 
	- 模块间的接口是简单的。
- 模块规模适中：
	- 模块数目增加，每个模块的规模将减小，开发单个模块的成本减少；模块之间关系的复杂程度会增加，设计模块间接口所需要的工作量也将增加。 
	- 通常规定其语句行数为50~100行，最多不超过500行。
- 结构图的深度和宽度适中 
- 结构图中扇入扇出适当 (3~4为宜，尽量不要超过8）
- 模块的作用域应在控制域之内
	- 模块的控制域是指它本身及其所有的从属模块的集合。 
	- 模块的作用域是指受其判定影响的所有模块的集合。
- 模块功能的完整性：
	- 执行规定的功能的部分。
	- 出错处理的部分：回送出错标志
	- 结束处理的部分：返回/处理一系列数据后，给调用者返回“结束标志”。 
- 消除重复功能：
	- 完全相似，考虑合并。
	- 局部相似，考虑独立出来重新定义成一个下层模块。

## 详细设计
### 软件设计原则
- 开闭原则 OCP: 
	- 软件实体对扩展开放，对修改关闭。 
	- 对扩展开放，意味着有新的需求或变化时，可以对现有代 码进行扩展，以适应新的情况。
	- 对修改关闭，意味着尽量避免对已有代码进行修改。
- 里氏替换原则 LSP: 
	- 程序中的父类都可以正确的被子类替换。 
	- 子类可以替换任何父类出现的地方，并且经过替换后，代码还能正确工作
- 依赖倒置原则 DIP: 
	- 模块之间交互应该依赖抽象，而非实现。 
- 单一职责原则 SRP: 
	- 每个软件模块只应该负责一项职责。
	- 职责越单一，被修改的原因就越少，模块的内聚性就越高，被复用的可能性就越大，也更容易被理解。
- 接口隔离原则 ISP: 客户端对类的依赖基于最小接口。 
	- 使用多个专门的接口，而不是一个总接口。 
	- 不能强迫用户去依赖那些他们不使用的接口。
- 迪米特法则 LOD: 一个类对于其它类知道的越少越好。
	- 最小化、简单化模块间的通信，并达到松耦合的目的。
- 合成复用原则 CRP: 优先使用组合/聚合，而不是类继承。 
	- 优先使用对象的组合/聚合将有助于保持每个类被集中在单个任务上。类和类继承层次会保持较小规模。
### 软件设计模式
- ![[软件设计模式分类.png]]
- Bridge模式
	- 问题场景： 多维变化点
	- 实现方式： 
		- 抽象部分与实现部分分离 
		- 实现每个变化点的独立变化
- Adapter模式：
	- 问题场景： 
		- 现有接口和需要的接口不匹配 
		- 需要复用现有的类 
		- 需要与第三方库或服务进行集成 
		- 各部分接口不一致问题
	- 实现方式：
		- 接口转换，增加适配器作为中间件
- Factory模式：
	- 问题场景： 
		- 对象的创建和初始化逻辑复杂 
		- 对象的创建具有变化的可能性 
		- 多个相关对象的创建
	- 实现方式：
		- 定义一个创建对象的接口-工厂类， 负责实例化过程
- Facade模式：
	- 问题场景： 
		- 简化复杂系统内部细节 
		- 解耦客户端与各子系统，提供简单的接口 
		- 提供系统的安全访问
	- 实现方式：
		- 增加一个总接口用于交互
- Command模式：
	- 问题场景： 
		- 请求的发送者和接收者解耦 
		- 请求需要方便地被序列化、存储、传递和撤销 
		- 支持命令的动态配置和替换
	- 实现方式：
		- 抽象命令接口Command（法典）：声明执行的方法。 
		- 具体命令对象ConcreteCommand（具体圣旨）：具体的命令。
# 软件测试
## 基本概念
### 软件错误
- 软件缺陷：
	- 存在于软件中的不希望的或不可接受的偏差。 
	- 程序本身的特性，以静态形式存在于软件内部。 
	- 当软件运行至某一特定条件时，软件缺陷被激活
- 软件故障：
	- 软件运行过程中出现的一种不希望或不可接受的内部状态。 
	- 动态行为，是软件缺陷被激活的表现形式
- 软件失效：
	- 软件运行时产生的一种不期望或不可接受的外部行为结果。 
	- 系统运行行为对用户要求的偏离。
### 软件测试的目的和原则
- 目的：尽可能多地发现软件中的错误、排除错误
- 原则：
	- 尽早地和不断地进行软件测试。 
	- 测试用例=测试输入数据和+预期输出结果。 
	- 应由第三方人员从事测试工作。 
	- 测试用例应当包括合理的输入条件和不合理的输入条件。 
	- 注意测试中的错误群集现象。 
	- 严格执行测试计划，排除测试的随意性。 
	- 妥善保存测试计划、测试用例、出错统计和最终分析报告
### 测试与质量的关系
- 软件质量是生产出来的，不是测试出来的。 
- 软件的验证和确认需要测试的支持。
